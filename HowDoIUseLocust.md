    <article class="markdown-body entry-content" itemprop="text"><h1><a id="user-content-how-do-i-locust" class="anchor" href="#how-do-i-locust" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>How do I Locust.</h1>

<p>Locust generates load and does pretty much nothing else. It comes with a web
server with UI controls out of the box. It can also run in a single-master,
multiple-slave configuration in which the slaves generate load and the master
aggregates reports from the slaves.</p>

<p>You write a locust file, which is just regular python file</p>

<pre><code># a descriptively named locust file
something.py
</code></pre>

<p>Put a task set in your locust file. Each task typically contains a particular
API operation.</p>

<pre><code>import locust
class MyTaskSet(locust.TaskSet):

    @locust.task
    def do_get(self):
        self.client.get('/resources/{id}')
</code></pre>

<p>Then define other test parameters, including the api endpoint to hit and the wait time:</p>

<pre><code>class MyLocust(locust.HttpLocust):
    task_set = MyTaskSet

    min_wait = 900
    max_wait = 1100

    host = 'http://localhost:8000'
</code></pre>

<p>The min/max wait times control the amount of time each simulated user waits
between executing tasks. Each user will execute a task at random, wait a random
amount of time between <code>min_wait</code> and <code>max_wait</code>, and then repeat.</p>

<p>At this point, you can start locust's web server:</p>

<pre><code>locust -f something.py
</code></pre>

<p>Then you can start the test in one of two ways:</p>

<ol>
<li>Go to <code>localhost:8089</code> in your browser and type in numbers and click start.</li>
<li>POST <code>localhost:8089/swarm</code> with <code>{"locust_count": 3, "hatch_rate": 1}</code></li>
</ol>

<p>The locust count is the total number of users to spawn. The hatch rate is the
number of users to spawn per second, starting from zero when load generation
first begins. (The "hatching" period is the rampup period from when the test
first starts until the max number of users is reached.)</p>

<p>Each user does the following:</p>

<ol>
<li>Pick one of the tasks from your locust file</li>
<li>Run the task (execute that task function)</li>
<li>Pick a random wait time between min_wait and max_wait (specified in your locust file)</li>
<li>Wait that amount of time</li>
<li>Repeat from 1</li>
</ol>

<h3><a id="user-content-but-how-do-i-slavery" class="anchor" href="#but-how-do-i-slavery" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>But how do I slavery??</h3>

<p>Locust can be run in a single-master, multiple-slave configuration. The slaves
do all the load generation, while the master controls and monitors. Every (by default)
3 seconds, a slave sends a single report for all requests made <em>on that slave</em>
in the last 3 seconds. The master receives these reports from all of its slaves
and consolidates them in real time.</p>

<p>The master controls the starting and stopping of <em>load generation</em> on the slaves.
The master cannot start/stop the locust process running on the slaves. This
means you need to create servers and start locust's processes yourself.</p>

<p>To start the master process:</p>

<pre><code>locust -f something.py --master
</code></pre>

<p>You must start the master process before the slaves. Then start the slaves:</p>

<pre><code>locust -f something.py --slave --master-host=&lt;master-ip&gt;
</code></pre>

<p>You should be able see the clients connect/disconnect in the master's logs.</p>

<h3><a id="user-content-random-tips" class="anchor" href="#random-tips" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Random tips</h3>

<h5><a id="user-content-detecting-whether-youre-the-master-or-slave" class="anchor" href="#detecting-whether-youre-the-master-or-slave" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Detecting whether you're the master or slave</h5>

<p>It seems normal to use the same locust file on the master and slave (I've never
tried using a master locust file and a separate, different slave locust file. Even
if it works, you lose the ability to run your tests in non-distributed mode.)</p>

<p>But sometimes I wanted to do certain things only on the master (like saving
a report to disk) and some things only on the slave (like doing some data prep
before starting load generation).</p>

<p>From what I can tell, locust doesn't provide a way to detect which mode you're
running in. What I did was check for the command line arguments:</p>

<pre><code>import sys

def is_master():
    return '--master' in sys.argv

def is_slave():
    return '--slave' in sys.argv
</code></pre>

<h5><a id="user-content-event-hooks" class="anchor" href="#event-hooks" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Event hooks</h5>

<p>Locust has some nifty event hooks that let you execute a function when that event is fired. Locust has <a href="http://docs.locust.io/en/latest/api.html#available-hooks">some built-in events</a>, or you can create your own:</p>

<pre><code>import locust.events

my_event = locust.events.EventHook()

def handler1(a, b): print "handled", a, b
def handler2(a, b): print "sum", a + b

my_event += handler1
my_event += handler2

# invokes each event handler in the order you inserted them
# note that you have to use keyworded arguments when you call .fire()
my_event.fire(a=1, b=2)
</code></pre>

<h5><a id="user-content-pre-test-actions" class="anchor" href="#pre-test-actions" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Pre-test actions</h5>

<p>There are several different points at which you can do something before the test starts. Some of these are kind of non-obvious, and I ran into situations where a setup task was being run every time a new user was spawned (during rampup) instead of just when load generation started. Anyway, these are the points at which you can do something before/after your test:</p>

<ol>
<li><p><em>At process startup</em>: You can do whatever you like in the global scope of your locust file, but this will only occur once for the entire time your locust process is running. I read config files and do other "framework" setup, like registering event handlers or preparing integration with another service.</p>

<pre><code>import locust
# read the config file once, at the start of the locust process
CONF = get_config()
# only register this event handler once
locust.events.locust_start_hatching += get_auth_token

class MyTasks(locust.TaskSet):
    ...
</code></pre></li>
<li><p><em>At test start</em>: You click the start button to start load generation You can hook into this by attaching event handlers to the  <code>locust.events.locust_start_hatching</code> event:</p>

<pre><code>def do_thing():
    # do a thing
locust.events.locust_start_hatching += do_thing
</code></pre>

<p>This will call <code>do_thing</code> exactly once when you press the start button. If you're running locust in master/slave mode, then <code>locust.events.locust_start_hatching</code> fires only on slaves, and <code>locust.events.master_start_hatching</code> fires only on the master.</p></li>
<li><p><em>At user spawn time</em>: You can do per-user setup in two places. Locust will call the <code>on_start</code> method when a Locust user is started:</p>

<pre><code>class MyTasks(locust.TaskSet):
    def on_start(self):
        # Each locust user gets a different id
        self.random_id = str(uuid.uuid4())
</code></pre>

<p>Locust creates a new instance of your TaskSet class once per user, so you can also do setup in the class constructor (I think this is less preferred):</p>

<pre><code>class MyTasks(locust.TaskSet):
    def __init__(self, *args, **kwargs):
        super(MyTasks, self).__init__(*args, **kwargs)
        # Each locust user gets a different id
        self.random_id = str(uuid.uuid4())
</code></pre></li>
</ol>

<h5><a id="user-content-grouping-requests-in-the-report" class="anchor" href="#grouping-requests-in-the-report" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Grouping requests in the report</h5>

<p>By default, locust will use <code>/resources/{uuid}</code> as the name that shows up
in the summary report. But if your the url has an id in it, you'll get a separate entry
in locust's report for each id.</p>

<p>Instead, you can provide an explicit <code>name</code> argument for Locust to use in its report:</p>

<pre><code>@locust.task
def do_get(self):
    self.client.get('/resources/{id}', name='/resources/UUID')
</code></pre>

<h5><a id="user-content-manually-marking-requests-as-successfailure" class="anchor" href="#manually-marking-requests-as-successfailure" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Manually marking requests as success/failure</h5>

<p>Normally, locust automatically records a success or failure by looking at the
http status code. You can use a <code>with</code> block to override this:</p>

<pre><code>with self.client.post('/things', catch_response=True) as post_resp:
    if post_resp.json()['status'] == 'ACTIVE':
        post_resp.success()
    elif post_resp.json()['status'] == 'ERROR':
        post_resp.failure("Saw ERROR status")
    else:
        post_resp.failure("Unknown error")
</code></pre>

<h5><a id="user-content-achieving-precise-request-rates" class="anchor" href="#achieving-precise-request-rates" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Achieving precise request rates</h5>

<p>You can compute the approximate request rate using the average wait time and
the total number of users. This will be inaccurate though, because it doesn't
take into account the time each user spends executing a task. This is a problem
when you have long running tasks (e.g. if you need to poll).</p>

<p>For example, consider a long running task like this:</p>

<pre><code>@task
def do_sleep(self):
    time.sleep(60)
</code></pre>

<p>A locust user will enter this task and sleep for 60 seconds. If you have a small
number of users, chances are they'll all eventually get stuck sleeping in this task,
unable to continue executing other tasks, so your request rates will
plummet. (Due to reasons, you should actually use gevent.sleep() instead of
time.sleep(). See the gevent docs for more.)</p>

<p>Instead, you want the task to take as short a time as possible. To do this, you
can spawn a new greenlet that sleeps in the background:</p>

<pre><code>def _do_async_thing_handler(self):
    gevent.sleep(60)

@task
def do_async_thing(self):
    gevent.spawn(self._do_async_thing_handler)
</code></pre>

<p>This accomplishes two things:</p>

<ol>
<li>Your task (do_async_thing) returns effectively immediately, and the
locust user can go on to do other things. This means your actual request
rate should much be closer to what you expect.</li>
<li>Your task's functionality (_do_async_thing_handler) continues running in
the background and will terminate on its own.</li>
</ol>

<h5><a id="user-content-polling-for-things-asynchronously" class="anchor" href="#polling-for-things-asynchronously" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Polling for things asynchronously</h5>

<p>You can use a similar pattern as above to poll for an active status. The only
thing to watch out for here is having too many greenlets running in the
background on a single box (you can lower the polling frequency or run in
distributed mode to solve this):</p>

<pre><code>def _do_async_thing_handler(self):
    # using `with` prevents locust from making an entry in its report
    with self.client.post('/things', catch_response=True) as post_resp:
        id = post_resp.json()['id']

        # Now poll for an ACTIVE status
        end_time = time.time() + timeout
        while time.time() &lt; end_time:
            r = self.client.get('/things/' + id)
            if r.ok and r.json()['status'] == 'ACTIVE':
                post_resp.success()
                return
            elif r.ok and r.json()['status'] == 'ERROR':
                post_resp.failure("Saw ERROR status")
                return

            # IMPORTANT: make sure you yield to other greenlets by using
            # gevent.sleep. Otherwise, you'll block the world.
            gevent.sleep(1)
        post_resp.failure("Polling timed out")

@task
def do_async_thing(self):
    gevent.spawn(self._do_async_thing_handler)
</code></pre>

<p>The key things here are:</p>

<ol>
<li>Spawning a new greenlet so your locust users don't block when running
your tasks, so your request rate is unaffected.</li>
<li>Using the <code>with</code> block to let us control the success condition</li>
<li>Using gevent.sleep in your polling loop to avoid blocking the world</li>
<li>Calling .success() or .failure() to mark the original request as a
success or failure. Locust will then compute and store the duration from the time
the request was made until the time the .success() function was called.</li>
</ol>

<p>(There's another minor issue here... When you stop the test, any greenlets
running in the background will continue running. I tracked my greenlets in a
list and used one of locust's event hooks to kill them all when the test is stopped)</p>
</article>
